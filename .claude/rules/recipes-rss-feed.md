# Recipes RSS Feed

The recipes RSS feed at `/recipes/index.xml` is generated by `layouts/recipes/list.rss.xml` at Hugo build time from `data/recipes.yaml`. Every entry in the yaml must carry a non-empty `added` (ISO 8601 date, `YYYY-MM-DD`) for it to appear in the feed — empty-string `added` is silently dropped.

## Why

The RSS template derives each item's `<pubDate>` from the recipe's `added` field via `time.AsTime`. That call errors at build time on an empty string, so the template applies a defensive filter — `where "added" "ne" ""` — and skips any row that would otherwise crash it. The failure is silent: the build succeeds, the feed generates without the affected recipe, and the only signal that something is wrong is a subscriber not seeing an entry they expected. That silence is the failure class this rule guards against.

The failure originates in `scripts/sync-recipes.py` when git's `--follow --diff-filter=A --reverse` combination can't trace a recipe's first-add commit through its rename history (a known git behavior, not a bug in the sync script's git invocation). Without a fallback, the sync writes `added: ""` and the recipe drops from the feed. The sync script now guards against this via a fallback chain — existing yaml value → today's date — but a future refactor could accidentally strip the guard.

## The rule

- Every entry in `data/recipes.yaml` must have a non-empty `added` string in ISO 8601 date form (`YYYY-MM-DD`).
- Never hand-write `added: ""` in `data/recipes.yaml`. If a recipe genuinely has no known add date, use today's date and note the choice in the commit message.
- `scripts/sync-recipes.py` must maintain a fallback chain for the case where `_git_first_added` returns empty: preserve the existing yaml value for the same `path`, or fall back to today's date. Log a stderr line explaining which branch fired so the sync run is auditable.
- `layouts/recipes/list.rss.xml` must keep the defensive `where "added" "ne" ""` filter. Removing it exposes `time.AsTime` to empty strings and crashes the entire feed build.

## Signals

| Signal | Fix |
|---|---|
| A row in `data/recipes.yaml` has `added: ""` | Set to a real ISO date. If unknown, use today. Log the intent in the commit message. |
| `scripts/sync-recipes.py` writes `added: ""` for a recipe | The fallback chain in `main()` failed. Check that the existing-yaml lookup and the `datetime.date.today().isoformat()` branch are both intact. |
| A subscriber reports a recipe missing from `/recipes/index.xml` after a sync | Grep `data/recipes.yaml` for `added: ""`. Every match is an entry silently dropped from the feed. |
| Diff to `layouts/recipes/list.rss.xml` removes the `where "added" "ne" ""` filter | Restore it. Without the filter, one empty `added` crashes `time.AsTime` and the whole feed fails to build. |
| Diff to `scripts/sync-recipes.py` deletes the empty-`added` fallback branches | Restore both branches — existing-yaml preservation *and* today-date fallback. One without the other is not equivalent. |

## When this rule fires

- Any `Write` or `Edit` to `data/recipes.yaml`.
- Any `Edit` to `scripts/sync-recipes.py` that touches the `added` computation, the `_git_first_added` function, `_load_existing_added_map`, or the fallback branches in `main()`.
- Any `Edit` to `layouts/recipes/list.rss.xml`.
- Any invocation of `/sync-recipes` that emits `no add date from git` to stderr.

## Report as

- **Must Fix** — a diff to `data/recipes.yaml` introduces or preserves `added: ""`. The affected recipe silently drops from `/recipes/index.xml`.
- **Must Fix** — a diff to `scripts/sync-recipes.py` removes the empty-`added` fallback branches without replacing them. Every future rename in the recipes repo could silently drop a recipe from the feed.
- **Must Fix** — a diff to `layouts/recipes/list.rss.xml` removes the defensive `where "added" "ne" ""` filter. One empty entry would crash the entire feed build.
- **Should Fix** — the sync script's `no add date` stderr line loses its context (which fallback branch fired, which recipe path). Silent healing is harder to audit than loud healing.

## Not covered by this rule

- **Music RSS feed** — the music section uses Hugo's built-in RSS template because its content adapter emits real Hugo pages per song. Its date invariant is enforced by the content adapter setting the page-level `.Date` from the yaml's `added`; see `content/music/_content.gotmpl` and the inline comment there for the `time.AsTime` conversion detail.
- **Category-page RSS feeds** — `content/recipes/appetizers.md`, `content/recipes/beef.md`, etc. each get their own auto-generated `/recipes/<category>/index.xml` from Hugo's default template. Those aren't affected by this rule — they're driven by content pages, not by `data/recipes.yaml` rows.
