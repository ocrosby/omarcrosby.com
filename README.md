# omarcrosby.com

Personal website for Omar Crosby, built with Hugo and deployed to Fly.io.

![CI](https://github.com/ocrosby/omarcrosby.com/actions/workflows/ci.yml/badge.svg)
![Release](https://github.com/ocrosby/omarcrosby.com/actions/workflows/release.yml/badge.svg)

## Overview

A static site generated by [Hugo](https://gohugo.io) using the `puppet` theme, served from a small `nginx:alpine` container on [Fly.io](https://fly.io). Releases are automated: Conventional Commits pushed to `main` are analyzed by [`jedi-knights/go-semantic-release`](https://github.com/jedi-knights/go-semantic-release), which tags a new `vX.Y.Z`, cuts a GitHub Release, and triggers a Fly deploy — all in one workflow.

## Features

- Static Hugo site (extended v0.160.1), theme via git submodule
- Multi-stage Docker build: Hugo builds inside the image, no CI-supplied artifacts required
- Release-driven CI/CD: push a `feat:` or `fix:` commit → automated version bump, tag, GitHub Release, and Fly deploy
- PR quality gates: Hugo build validation, broken link check, Markdown lint, typo scan
- Apex canonical (`omarcrosby.com`), `www` 301-redirects via nginx

## Requirements

- [Hugo extended](https://gohugo.io/installation/) `>= 0.101.0, < 0.128.0` — theme uses `resources.ToCSS`, removed in Hugo 0.128+
- [Docker](https://www.docker.com/) `>= 20.10` (for container builds)
- [flyctl](https://fly.io/docs/hands-on/install-flyctl/) `>= 0.3` (for deploys)
- Git with submodule support

## Installation

```bash
git clone --recurse-submodules https://github.com/ocrosby/omarcrosby.com.git
cd omarcrosby.com
```

If cloned without submodules, initialize the theme:

```bash
git submodule update --init --recursive
```

## Usage

Preview locally with Hugo's dev server:

```bash
hugo server -D
```

Then open <http://localhost:1313>. Draft content (`draft: true` in front matter) shows only with the `-D` flag.

Build and preview the production container:

```bash
docker build -t omarcrosby-com .
docker run --rm -p 8080:8080 omarcrosby-com
```

Then open <http://localhost:8080>.

## Configuration

| File | Purpose |
|------|---------|
| `config.toml` | Hugo site configuration (base URL, theme, menu, params) |
| `fly.toml` | Fly.io app configuration (region, machine size, port) |
| `.semantic-release.yaml` | Conventional-commit release rules and tag format |
| `.markdownlint.yaml` / `.markdownlintignore` | Markdown lint rules and exclusions |
| `.lycheeignore` | Broken-link-check URL exclusions |
| `typos.toml` | Typo-scan configuration |

Required GitHub Secret for deploys: `FLY_API_TOKEN` (create with `flyctl tokens create deploy --app omarcrosby-com`).

## Development

Add new content:

```bash
hugo new content/posts/my-post.md
```

Run the same quality gates CI runs, locally:

```bash
hugo --minify --gc --panicOnWarning
npx markdownlint-cli2 "**/*.md" "#themes" "#public" "#node_modules"
typos
lychee './public/**/*.html'
```

Deploy manually (bypassing the release workflow):

```bash
flyctl deploy --remote-only
```

## Contributing

This is a personal site, but PRs and issues are welcome. Use [Conventional Commits](https://www.conventionalcommits.org) — `feat:`, `fix:`, `docs:`, `chore:`, etc. — since the release workflow keys off commit types.
